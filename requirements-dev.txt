# Development / CI tooling for the multi-model work.
# Pinned so a new tool release can't turn CI red without an intentional bump
# (e.g. mypy 2.x's stricter checks, or a new ruff rule set). Bump deliberately.
# Runtime deps (numpy, opencv-python, scipy) stay UNpinned in pyproject.toml —
# a library should not over-constrain its consumers. The ONE exception is numpy
# below: its type stubs feed `mypy ds_msp/core`, so an unpinned numpy release can
# turn CI red with zero code changes (numpy 2.5.0, published 2026-06-21, broke the
# typed-core check this way). An upper bound — not an exact pin — because the test
# matrix spans py3.10 (numpy dropped 3.10 in 2.3, so it resolves 2.2.x there)
# through py3.12 (resolves 2.4.x). `<2.5` excludes the stub-breaking release while
# letting each Python pick its latest compatible wheel. Dev/CI only; pyproject
# keeps numpy unpinned for consumers. Lift the bound once core is verified vs 2.5.
pytest~=9.1
pytest-cov~=7.1
mypy==2.1.0
ruff==0.15.18
import-linter==2.11
numpy<2.5
