# DS-MSP living-book blueprint

> **Status: architecture spec (pilot phase).** This file is the editorial plan for turning
> `docs/` into a Diátaxis-organized "living book" served by MkDocs Material. It defines the
> information architecture, the page→file→mode map, the single-source-of-truth policy, the
> duplication to collapse, the MkDocs nav/plugin spec, the flagship Chapter 8 spec, the README
> redesign, and the ordered chapter backlog. **It contains no chapter prose** — every chapter is
> written by the `/write-chapter` pipeline and gated by `leak-guard`.

---

## 1. The book in one sentence

A clean, tested wide-FOV camera library (`ds_msp/`) **and** a guided course in the geometry
behind SLAM/SfM, where every chapter proves a number on real public data.

The book is the union of five exemplar patterns:

| Borrowed pattern | From | Where it lands here |
| :-- | :-- | :-- |
| Thin front door → install → one MWE → "where next" | FastAPI / Typer README | the README redesign (§7) |
| Ordered learning path, each step needs only what came before | The Rust book | Tutorials section / `docs/learn/` (§2) |
| Executable, number-proving examples | Typer (testable docs) | every tutorial anchored to `examples/NN_*.py` (§2) |
| Rigorous derivations kept *out of* the tutorial flow | slambook2 | Explanation section / `docs/explain/` (§2) |
| Strict separation of concerns; library stays terse | SAD (*SLAM in Autonomous Driving*) | the four-mode split + "teaching lives in docs/, never in `ds_msp/`" rule |

```
    ├── tier1_implementation_spec.md
    └── …
```

**Mode boundaries enforced editorially:**

- A **tutorial** (`docs/learn/`) never stops to derive; it *uses* the result and links to
  `docs/explain/` for the why. The example it runs prints the number it claims.
- An **explanation** (`docs/explain/`) never holds a "do this then that" recipe; it proves
  claims with named tests and links to the tutorial that exercises them.
- A **how-to** (`docs/how-to/`) is a single task, start to finish, no theory — it links out to
  the explanation for the why and to reference for the API.
- **Reference** is API facts only, generated by `reference-writer` via mkdocstrings.

> Today the `docs/learn/` deep-dives blend Tutorial + Explanation. The IA splits them: the
> *runnable, number-proving* half stays a tutorial; the *derivation* half moves to `docs/explain/`.
> See the duplication-collapse list (§5) for the exact moves.

## 4. Single-source-of-truth policy

One canonical home per topic. Everything else **links**, never restates.

| Topic | Single source of truth | Everyone else does | 
| :-- | :-- | :-- |
| Roadmap / build order | `docs/ROADMAP.md` | links to it; no roadmap prose elsewhere |
| Curriculum / learning path | `docs/learn/README.md` | links to it; README shows at most a 2-line teaser + link |
| Multi-model usage (cookbook) | `docs/how-to/*` (extracted from `MULTI_MODEL.md`) | README links to the relevant how-to |
| API facts | `docs/reference/*` (mkdocstrings) | tutorials/how-tos link to a reference page, never re-table the API |
| Two-view geometry math | `docs/explain/two_view_geometry.md` | Ch.8 tutorial + `two_view.py` docstring link to it |
| FOV / validity math | `docs/explain/projection_validity_and_fov.md` | README + Ch.3 link to it |
| Accuracy/verification numbers | the test + the chapter that proves it | README quotes ≤1 headline number, links to source |
| Writing standard | `docs/WRITING_GUIDE.md` | all docs follow; no restated style rules |
| Doc-excellence rubric | `docs/research/doc_excellence_spec.md` | pipeline references it |

**Rule of thumb the editor enforces:** if the same fact (a number, a curriculum table, a roadmap
item) appears in two served files, one of them is wrong the moment the other changes. Collapse to
one + link.

## 5. Duplication to collapse

The **roadmap/curriculum 3× duplication** confirmed in the repo, plus the other repeats:

### A. The curriculum table — duplicated 3×
1. **`README.md`** lines ~191–215 ("Learn: the guided curriculum") restate the full Part I table,
   the Part II table, and the Part I/II narrative.
2. **`docs/learn/README.md`** — the canonical curriculum index (Part I + Part II tables, mermaid).
3. **`docs/ROADMAP.md`** "Next (learning curriculum)" (lines ~29–41) restates Ch.2–7 with anchors.

**Collapse →** `docs/learn/README.md` is the single source.
- README "Learn" section shrinks to a 2-sentence teaser + one link to `docs/learn/README.md`
  (the per-chapter table is deleted from the README).
- ROADMAP "Next (learning curriculum)" drops the per-chapter restatement; it keeps *build-order
  priority only* and links to `docs/learn/README.md` for the chapter list.

### B. The Part I / Part II narrative — duplicated 3×
The "two arcs: Part I Calibration / Part II Geometry & 3D; library shipped & tested, chapters
landing" paragraph appears verbatim-ish in **README** (~196–215), **`docs/learn/README.md`**
(~37–44, 81–87), and **ROADMAP** Tier-1 (~103–108).

**Collapse →** `docs/learn/README.md` owns the arc narrative. README and ROADMAP each keep one
sentence + link.

### C. Roadmap pointer — repeated everywhere
README links to ROADMAP from "Repository map", "Learn", "Part II", and "Roadmap" sections (4×).

**Collapse →** one "Roadmap" section in README links to ROADMAP once; other mentions drop the link.

### D. Multi-model usage — duplicated README ↔ MULTI_MODEL.md
README "Using the library" (create/project/undistort/PnP/convert/LDC, ~219–329) re-teaches what
`docs/MULTI_MODEL.md` already documents at length.

**Collapse →** the per-task content becomes the **How-to cookbook** (`docs/how-to/*`).
`MULTI_MODEL.md` is split: task recipes → `docs/how-to/*`; the architecture/design-guarantees
section → `docs/reference/index.md`. README keeps only the single MWE (§7) and links to the
cookbook.

### E. FOV / validity essay — wrong mode in README
README "Deep dive: FOV, validity & undistortion" (~378–447) is an *Explanation* essay living in
the front door, and it overlaps Ch.3 + the unprojection equations already in the DS docstrings.

**Collapse →** move to `docs/explain/projection_validity_and_fov.md` (Explanation). README keeps
one line ("a fisheye sees >180°, which a pinhole can't hold — why →") linking to it. Ch.3 also

`docs/research/mvg_two_view_geometry.md` is a polished Explanation that currently lives in the
*design-records* dir.

**Collapse →** relocate to `docs/explain/two_view_geometry.md` (it *is* the Explanation page);
leave a one-line pointer in research if any old link needs it. `two_view.py`'s docstring pointer
updates to the new path.

### G. Accuracy numbers — duplicated README ↔ ROADMAP ↔ tests
Calibration accuracy (`0.003% focal / 0.08 px`), stereo (`0.062° / 0.25 mm`), test count
(`237`) appear in README, ROADMAP, learn/README, and the capstone. These have a history of going
stale together (observed multiple times).

**Collapse →** each number has one owner: the chapter/test that produces it. README/ROADMAP quote
at most the single headline (`0.003% focal`) with a link; the test count badge is generated, not
hand-typed in prose. (Editorial note for the pipeline, not a blocker for the pilot.)

## 7. Chapter 8 spec — "Two-view geometry on rays" (PILOT)

The flagship chapter and the entry point to Part II. It must read as a Rust-book chapter: one
idea per step, each step prints a verifiable number, theory linked out (not inlined).

| Field | Value |
| :-- | :-- |
| **Title** | Two-view geometry on rays |
| **Diátaxis mode** | Tutorial |
| **Target file** | `docs/learn/08_two_view_geometry_on_rays.md` |
| **Code anchors** | `ds_msp/mvg/two_view.py`, `ds_msp/mvg/ransac.py` |
| **New example** | `examples/10_two_view_pose_tumvi.py` (real-data) — see anchoring note below |
| **Prerequisites** | Ch.1 (project/unproject are inverses), Ch.3 (why `z>0` is the wrong cheirality test for wide-FOV). Reader has run the dataset fetcher. |
| **Theory lives in** | `docs/explain/two_view_geometry.md` (relocated `mvg_two_view_geometry.md`) — the chapter links to it for proofs; it does **not** inline the epipolar derivation. |

### The number the chapter proves
The chapter proves **two** numbers, both already test-backed (no invented figures):

1. **Documented synthetic DS round-trip (primary, deterministic).**
   3D points → `DoubleSphereModel.project` into two views → `unproject` back to bearing rays →
   `recover_pose` → compare to the ground-truth `(R, t)`. The recovered pose matches ground truth
   to **rotation error < 1e-3°** and **translation-direction error < 1e-3°**, "limited only by
   the project/unproject round-trip." This is exactly
   `tests/mvg/test_two_view.py::test_recover_pose_through_a_real_double_sphere_camera`
   (`DoubleSphereModel(fx=300, fy=300, cx=320, cy=320, xi=0.3, alpha=0.6)`), so the chapter's
   headline number is asserted in CI and cannot drift. **This is the chapter's required number.**

2. **Robustness on outliers (RANSAC, deterministic).**
   With 30% of correspondences corrupted, `ransac_relative_pose` recovers the pose to
   **rotation < 0.5°**, **translation-direction < 2.0°**, with **inlier precision > 0.95 and
   recall > 0.9**, where the naïve eight-point lands several degrees off — from
   `tests/mvg/test_ransac.py::{test_ransac_recovers_pose_under_30pct_outliers,
   test_ransac_beats_naive_eight_point_with_outliers}`.

### Real public-data anchoring (the "on real data where applicable" requirement)
A real two-view pair is added on top of the synthetic proof, using machinery already shipped and
validated in `examples/09_monocular_vo_tumvi.py` (KLT tracks on **TUM-VI** cam0 + `load_kalibr`):

- Take two TUM-VI `room1` frames, KLT-match, `unproject` both pixel sets through the loaded
  `DoubleSphereModel` to bearing rays, run `ransac_relative_pose`.
- Report the **angular Sampson inlier residual** (radians, FOV-independent) and **inlier count /
  ratio** as the real-data number, and sanity-check the recovered translation direction against
  the consecutive-pose direction from TUM-VI mocap ground truth (the same GT loader `examples/09`
  already uses). The deterministic *correctness* claim stays the synthetic round-trip (number 1);
  the real-data pair demonstrates the pipeline runs end-to-end on a fisheye stream.

> Honest-status note for the writer: the synthetic DS round-trip is the verifiable headline; the
> TUM-VI pair is a real-data demonstration, not a sub-1e-3° guarantee (real correspondences carry
> noise). Frame it that way — do not over-claim a real-data pose accuracy the tests don't assert.

### Chapter shape (Tutorial, examples-first)
1. **Hook + MWE first.** Smallest runnable thing: 8 ray pairs in, `(R, t)` out, one number printed
   (`recover_pose` on synthetic rays) — before any theory.
2. **Why rays, not pixels.** One paragraph; the epipolar constraint *holds* on bearings; link to
   `explain/two_view_geometry.md` §1 for the proof.
3. **Eight-point on rays** (`essential_from_rays`) → run it, print residual `< 1e-10`.
4. **Decompose + cheirality** (`recover_pose`) → the >180° callback to Ch.3 (positive depth along
   the ray, not `z>0`). Print the synthetic round-trip number (**< 1e-3°**).
5. **Make it robust** (`ransac_relative_pose`) → corrupt 30%, show naïve fails, RANSAC recovers;
   print the **< 0.5°** number + inlier precision/recall.
6. **On real data** → the TUM-VI pair via `examples/10`; print inlier ratio + angular residual.
7. **Try it yourself / Next** → change the rotation magnitude / outlier fraction and predict;
   link to Ch.9 (manifold optimization) as the next step.
